# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

This is **Gearent** (also referenced as "Zewalo" in older planning docs), a rental/warehouse management SaaS built on Laravel 12 + Filament 4. It serves two distinct audiences from one codebase:

- **Admin panel** at `/admin` — built with Filament 4 (Resources, Pages, Clusters, Widgets). Handles rentals, inventory, customers, finance, CMS.
- **Customer storefront** (frontend) — traditional Blade views under `resources/views/frontend/**` with controllers in `app/Http/Controllers/` (not the `Frontend` subfolder alone; most are at the top level).

The app has a first-run **installation wizard**. `routes/web.php` gates *all* routes behind `File::exists(storage_path('installed'))` — if that marker file is missing, every request is redirected to `/setup`. When working on route/controller changes locally, ensure the `storage/installed` file exists or your routes will 404.

### Verifying edits in this local environment (read before "running" anything)

This dev box frequently has **no `vendor/` directory** (Composer deps not installed) and **no `storage/installed` marker**, so **anything that boots Laravel will NOT run locally** — `php artisan …`, `composer test`/`phpunit`, `pint`, `composer dev`, migrations, and `rentals:check-late --dry-run` all fail with `Failed to open … vendor/autoload.php`. **Do not run `composer install`/`composer update` just to verify an edit** — it triggers `filament:upgrade` (post-autoload-dump) and is a heavy, unrequested change. Treat artisan/test/migrate verification as **server-side / post-deploy** steps, and say so in your summary instead of pretending they ran.

PHP CLI is at **`C:\xampp\php\php.exe`** (not on `PATH`; the Bash tool's `php` won't resolve — use the full path via PowerShell). The XAMPP CLI also **lacks `ext-gd`**, so GD-dependent code (e.g. [`LabelImageService`](app/Services/LabelImageService.php)) can't be exercised locally either.

So the **correct local check for a code edit** is static, not runtime:
1. **Syntax**: `& 'C:\xampp\php\php.exe' -l path\to\File.php` on every changed `.php` file (Blade `@if/@foreach` won't parse with `-l`, but inline `@php … @endphp` blocks do).
2. **Exhaustiveness sweep**: when you add an enum/status value, grep for every `match($state|$status)` and hardcoded `@if/@elseif`/array map that switches on it (kanban/list blades, calendar `$colorMap`, storefront badges, `getStatus*` helpers) and confirm each either has a `default`/`@else`/`?? fallback` or gets the new case — a `match` with no `default` throws `UnhandledMatchError` at runtime. Prefer routing display through the `Rental::getStatus*()` helpers (which carry defaults) over re-inlining maps.
3. **Then** list the runtime commands (migrate, `rentals:check-late --dry-run`, `php artisan test --filter=…`) for the user/server to run after `composer install`.

## Architecture

### Product / inventory hierarchy

`Product` → `ProductVariation` (optional) → `ProductUnit` (serial-tracked physical unit) → `UnitKit` (accessory components attached to a unit).

- A product has *either* no variations (bare `product_id` composite key) *or* one-or-more variations (composite key `{product_id}:{variation_id}`). This distinction drives the `composite_id` format throughout the rental editor and all availability queries.
- `ProductUnit` is the schedulable atom — availability checks, `RentalItem`, and calendar events all operate at this level.
- `UnitKit` records accessory components (e.g. "battery pack A1") hanging off a `ProductUnit`. When `track_by_serial = true`, `UnitKitObserver::saving` calls `KitUnitLinker` to auto-resolve (or create) a shadow `ProductUnit` for the kit slot, stored as `linked_unit_id`. This keeps component serials independently trackable.
- `RentalValidationService` also enforces operational schedule (JSON setting `operational_schedule`) and holidays (JSON setting `holidays`). It is used for both rental and computer-booking date validation.
- `Product.buffer_time` (days) pads the post-return unavailability window to prevent back-to-back bookings without maintenance time.

### The Rental domain is the center of gravity

Everything revolves around the `Rental` model and its lifecycle: `quotation` → `confirmed` → `active` → `completed` / `partial_return` / `cancelled`. Additional statuses: `late_pickup` (confirmed, past start, not picked up), `late_return` (active, past end date), and `expired` (a **quotation** whose pickup date passed without ever being confirmed — a dead-end, like `cancelled`). Full constant list on `Rental`: `STATUS_QUOTATION`, `STATUS_CONFIRMED`, `STATUS_ACTIVE`, `STATUS_COMPLETED`, `STATUS_CANCELLED`, `STATUS_LATE_PICKUP`, `STATUS_LATE_RETURN`, `STATUS_PARTIAL_RETURN`, `STATUS_EXPIRED`.

- **Auto late/expiry transitions (only on date overrun — `quotation` is NOT a late-pickup source).** Three places must stay in sync: the scheduled `rentals:check-late` command ([CheckLateRentals.php](app/Console/Commands/CheckLateRentals.php), every 5 min), and `Rental::getRealTimeStatus()` (display) + `Rental::checkAndUpdateLateStatus()` (persist). The rules: `quotation` + `start_date < now` → **`expired`**; `confirmed` + `start_date < now` → **`late_pickup`**; (`active`|`partial_return`) + `end_date < now` → `late_return`. Historically the late-pickup trigger was `quotation` (a never-confirmed quote got auto-promoted to `late_pickup` and thus became pickup-able, skipping confirmation) — that was the bug this split fixed. `completed`/`cancelled`/`expired` are terminal (early-returned, never re-evaluated).
- **`expired` is non-blocking for availability** — it is deliberately absent from every active-status whitelist (the date-overlap queries in [ProductUnit.php](app/Models/ProductUnit.php), [CatalogController.php](app/Http/Controllers/CatalogController.php), `RentalEditor::callPrivateGetBookedUnitIds()`), so an expired quote's units free up automatically — exactly like `cancelled`. Quotation/confirmed rentals never set `ProductUnit.status` to RENTED anyway (that only happens at pickup), so expiring needs no unit refresh. `expired` **is** in `canBeEdited()` (unlike `cancelled`) — expiry is a soft timeout, so an expired quote can be re-dated / re-confirmed to revive it. The status helpers (`getStatusOptions/Color/HexColor/Label`, `getChecklistSteps`) all handle `expired` (gray, locked checklist like cancelled), so it propagates to the table filter, both status dropdowns, kanban, and the storefront badges. No data backfill on deploy — existing `late_pickup` rows from the old behaviour are ambiguous (no `confirmed_at` to distinguish) and are left for manual review.

Critical pieces:

- `app/Models/Rental.php` + `RentalItem` + `RentalItemKit` — rental with per-unit breakdown, including kit-level serial tracking.
- `app/Services/RentalValidationService.php` — double-booking prevention, product unit conflict resolution, date overlap checks. Any code that creates/edits rentals or checks availability should go through this service.
- `app/Observers/RentalObserver.php` — wired up in `AppServiceProvider::boot()`. Side-effects on rental state changes live here.
- `app/Http/Controllers/CheckoutController.php` (customer flow) and `app/Filament/Resources/Rentals/**` (admin flow) are the two entry points that create rentals; both must stay in sync on validation rules.
- `CatalogController::checkAvailability` powers live unit availability on the storefront — changes to availability logic must be mirrored in both the frontend calendar and this endpoint.

### Rental editor (Create / Edit) — custom Livewire, NOT Filament default

`/admin/rentals/create` dan `/admin/rentals/{id}/edit` **tidak memakai Filament form builder** seperti resource lain. Keduanya merender view `filament.rentals.editor` (Blade `resources/views/filament/rentals/editor.blade.php`) yang membungkus satu komponen Livewire: [`App\Livewire\Admin\RentalEditor`](app/Livewire/Admin/RentalEditor.php) dengan view [`resources/views/livewire/admin/rental-editor.blade.php`](resources/views/livewire/admin/rental-editor.blade.php) (~2000 baris, custom UI desktop + mobile).

- `CreateRental` page passes `record = null`; `EditRental` page passes existing `Rental` model. Komponen Livewire pakai `!$record || !$record->exists` untuk membedakan mode (tombol "Create Rental" / "Buat Rental" vs "Save Changes" / "Simpan"). Logika save: `Rental::create()` saat baru, `fill() + saveQuietly()` saat update.
- Items disimpan di array `$items` Livewire (state in-memory), **grouped per product-variation** (bukan 1 RentalItem = 1 baris UI). Setiap row punya `composite_id` (`{product_id}` atau `{product_id}:{variation_id}`), `quantity`, dan `unit_ids[]` (array serial unit IDs). DB-level `RentalItem` tetap satu-per-unit; mapping antara state Livewire ↔ DB ada di `RentalForm::syncRentalItems()` dan `RentalForm::groupItemsForForm()`.
- **`ItemsRelationManager` masih ada di codebase tapi TIDAK dirender** di halaman edit baru ini. Pengelolaan item sepenuhnya lewat Livewire editor. Jangan tambah action di RelationManager untuk fitur baru — tambahkan di `RentalEditor` + blade view.
- **Stock-aware tapi tidak menghalangi**: catalog popup (modal desktop + sheet mobile) tetap mengizinkan add saat stok 0 atau kurang dari qty diminta — row dibuat dengan slot `unit_ids` parsial / kosong, badge kuning "+N kosong" muncul di UI, dan user diarahkan ke fitur Transfer (lihat di bawah). Tombol `+` di catalog tidak pernah disabled.
- **Catalog desktop modal sinkron 2-arah**: tiap card produk menampilkan stepper `[− N +]` bila produk sudah ada di tabel, atau tombol `+` saja jika belum. Map qty dibangun dari `$items` tiap render sehingga catalog selalu cerminkan tabel rental. `decrementByComposite()` menghapus row saat qty turun ke 0.

### Cross-rental unit Move / Swap

Fitur untuk memindahkan / menukar / menarik `RentalItem` antar rental yang berbeda — solusi untuk kasus konflik booking: unit X sudah ter-booking di Rental A, lalu Rental B juga butuh unit X di periode overlap. Daripada "menolak karena stok 0", admin bisa **MOVE** unit dari A ke B, **SWAP** (X di A ↔ Y di B), atau **PULL** (tarik unit dari A untuk mengisi slot kosong di B yang sedang diedit).

- [`app/Services/RentalItemTransferService.php`](app/Services/RentalItemTransferService.php) — service inti dengan dua method publik: `move(RentalItem $sourceItem, Rental $targetRental)` dan `swap(RentalItem $itemA, RentalItem $itemB)`. Semua operasi dalam DB transaction. (Mode **PULL** di UI = `move()` terbalik: `move($itemDariRentalLain, $this->record)`.)
- **MOVE = transfer assignment unit, BUKAN pindah baris.** Yang dipindah adalah `product_unit_id`-nya, bukan `rental_id`:
  - **Rental sumber TETAP punya line produknya.** Source `RentalItem` di-null-kan `product_unit_id`-nya → jadi **ghost slot** (slot kosong). Quantity di sumber tidak berubah, hanya jumlah unit ter-assign yang turun. Produk **tidak pernah dihapus** dari tabel items. `product_id`/`product_variation_id` dipertahankan agar baris tetap merender produk yang sama.
  - **Rental tujuan**: kalau sudah ada ghost slot yang cocok (`product_unit_id` null, `product_id`/`variation` sama) → slot itu **diisi** (quantity TIDAK bertambah — ini kasus PULL: slot kosong "+N kosong" yang sudah dibuat user). Kalau tidak ada ghost slot cocok → buat baris baru = penambahan stok asli (quantity +1). Logika di `RentalItemTransferService::findGhostSlot()`.
  - Ini menjadikan MOVE simetris dengan SWAP (yang juga tukar `product_unit_id`, kedua baris tetap di rentalnya).
- **Guard status**: kedua sisi rental harus berstatus `QUOTATION`, `CONFIRMED`, atau `LATE_PICKUP`. Status `ACTIVE`/`LATE_RETURN`/`PARTIAL_RETURN`/`COMPLETED`/`CANCELLED` ditolak (lawan rental "belum dipickup" only). Konstanta `RentalItemTransferService::TRANSFERABLE_STATUSES`.
- **Recalc total**: tidak dilakukan manual di service. `RentalItem::$touches = ['rental']` otomatis trigger `RentalObserver::updated` saat item disimpan — observer yang handle subtotal/discount/tax/total lengkap. Karena MOVE sekarang menyimpan satu `RentalItem` di **masing-masing** rental (source jadi ghost, target dapat unit), kedua rental ter-touch otomatis — **tidak perlu** `$sourceRental->touch()` manual lagi (catatan: ghost slot sumber tetap punya subtotal `daily_rate × days`, jadi nilai sumber tidak berubah; sumber "masih memesan" produk, hanya butuh assign unit baru).
- **Konflik**: setelah operasi, service cek `checkAvailability()` di rental tujuan; jika muncul konflik baru, **seluruh DB transaction di-rollback** (`RuntimeException`).
- **Hooks otomatis**: saat `product_unit_id` berubah, `RentalItem::updated` hook auto-delete kit lama & attach kit baru; `RentalItem::saved` hook auto-relink parent/child kit. **Penting**: `attachKitsFromUnit()` sekarang early-return bila `productUnit` null (ghost slot tidak punya unit untuk diambil kitnya), dan reverse-check di `saved` hook di-skip saat `product_unit_id` null (kalau tidak, `where('unit_id', null)` jadi `whereNull` dan salah match).
- **Audit**: tidak ada activity log package. Service mencatat lewat `Rental::logActivity($message, $type)` → kolom JSON `rentals.activity_log` (bukan `notes`, agar catatan free-text operator tetap bersih). Tiap entri `{at, type, message, user}`, ditulis via `updateQuietly()` (tidak re-trigger observer / recalc total). Tipe yang dipakai: `move`, `swap`, `status`, `general`. Plus `Log::info('RentalItem MOVE/SWAP', [...])`.
  - **Card "Log Aktivitas"** dirender dari `activity_log` (newest-first) lewat partial [`partials/activity-log.blade.php`](resources/views/filament/resources/rentals/partials/activity-log.blade.php), di-`@include` di view-rental + RentalEditor (desktop & mobile, guard `@if($record && $record->exists)`).
  - Semua event audit harus pakai `logActivity()`, **jangan** append stempel ke `notes`. Sumber log saat ini: MOVE/SWAP (`RentalItemTransferService`), transisi status (`RentalObserver::updated`, otomatis untuk semua perubahan `status`), reopen Completed→Active + alasan (`RentalsTable`), late fee (`RentalsTable` & `InvoiceResource`), alasan pembatalan (`Rental::cancelRental`).
- **UI** di `RentalEditor`:
  - Method publik: `openMoveModal($unitId)`, `openSwapModal($unitId)` (per-serial), `openTransferForRow($itemKey, $mode)` (per row group — auto-pick unit jika row hanya punya 1 serial), `openPullModal($itemKey)` (tarik unit dari rental lain untuk isi slot kosong row ini).
  - Sebelum modal dibuka, `beginTransfer()` **auto-save** editor state via `persistInline()` agar source RentalItem benar-benar ada di DB. Setelah `confirmTransfer()` sukses, `redirect()` ke URL edit yang sama untuk reload state.
  - Tombol akses: (a) icon "transfer" (dua panah berputar) di row item desktop & mobile → dropdown "Move ke rental lain" / "Swap dengan rental lain"; (b) link kecil **Move · Swap** di tiap slot unit yang sudah ter-assign di Unit Modal. Hanya muncul kalau `canTransfer` true (rental berstatus eligible + record sudah exist).
  - Modal transfer (di blade view) menampilkan: picker unit (kalau dibuka dari row dengan >1 unit), select rental tujuan (filtered ke status eligible), dan select unit lawan (mode swap saja).

### Rental operations: Pickup and Return

Two custom Filament pages handle the physical handoff:

- `app/Filament/Resources/Rentals/Pages/PickupOperation.php` — marks rental `active`, records which items and kit accessories were handed out, optionally creates a `Delivery` record for outgoing items.
- `app/Filament/Resources/Rentals/Pages/ProcessReturn.php` — marks rental `completed` or `partial_return`, checks each item/kit back in, creates a return `Delivery`.

Both pages render custom Blade views (not Filament form pages) and implement `HasTable` for the items checklist. `Delivery` + `DeliveryItem` models track the logistics record for both directions.

#### Return settlement: late fee, invoice sync, manual add, reopen

The "Validate return" confirmation modal (`return-operation.blade.php`) runs the financial settlement. Key rules:

- **Late fee storage** — the only home is the column `rentals.late_fee`. The invoice's `late_fee`/`total` are **derived** (`Invoice::recalculate()` does `late_fee = Σ rentals.late_fee` and `total = Σ rentals.total`). There is **no separate late-fee table** and **no "auto-paid"** — a late fee just raises the rental/invoice total; collection happens via the normal invoice payment flow (**Finance → Accounts Receivable → Record Payment**, which writes a `FinanceTransaction` income and re-`recalculate()`s the invoice).
- **Late fee is computed by `Rental::calculateOverdueFee()`** from `Setting` keys `late_fee_mode` (`flat_per_day` / `per_unit_per_day` / `percentage_per_day` / `full_daily_rate` / `tiered`; legacy `late_fee_type` mapped for back-compat), `late_fee_amount`, `late_fee_tiers`, honoring per-product `Product.late_fee_daily_amount` overrides. `Rental::lateFeeBreakdown()` returns a human-readable rincian (mode, hours/days late, per-item lines) shown **open by default** in the settlement modal so the auto calc is transparent; its `fee` is taken straight from `calculateOverdueFee()` so the breakdown total always equals what's charged.
- **`ProcessReturn::validateReturn($data)`** resolves the late fee **once** (manual modal value wins over auto), passes it into **`Rental::validateReturn(?float $lateFee)`** which **honors that value (no silent re-overwrite)** and calls `recalculateTotal()` (so the completed total keeps tax + deposit, not the old `subtotal − discount + lateFee` shortcut).
- **Invoice sync after return** — `ProcessReturn::syncInvoiceAfterReturn()` delegates to **`Rental::syncOutstandingInvoice(string $noteContext)`**: if an invoice is linked it is `recalculate()`d (and reopened PAID→PARTIAL when a new balance appears); if **no invoice exists but a balance is now owed**, one is **issued** (mirroring the confirm-time flow: `RENTAL_INVOICE_ISSUED` journal + re-link existing DP/income via `Rental::rentalIncomeTransactions()`). Without this, a late fee on a never-invoiced rental would be invisible in Accounts Receivable.
- **Manual late fee** — the **"Add Late Fee"** action in the rental list's **Finance** action group ([`RentalsTable`](app/Filament/Resources/Rentals/Tables/RentalsTable.php)) has **no status restriction** (works on `completed` rentals) and now also calls `syncOutstandingInvoice('late fee')`. The separate **Add Late Fee on `InvoiceResource`** must write the fee to the **underlying rental** then `recalculate()` — writing it onto `invoice.late_fee` directly is wiped because that field is derived (this was a fixed bug).
- **Reopen a completed rental** — `Rental::reopenFromCompleted()` reverts `COMPLETED → ACTIVE` (clears `returned_date`, reopens the latest IN delivery to `DRAFT` + unchecks its items, runs `ProductUnit::refreshStatus()` so non-damaged units return to `RENTED`). Surfaced as the **"Reopen Rental"** row action (admin/super_admin only, requires a reason, audited to `notes` + `Log`). It is **operational only** — it does **NOT** reverse the prior completion's finance journals, so re-completing in advanced finance mode can double-post revenue (the modal warns; for just adding a missing late fee, prefer "Add Late Fee" over reopening). Only the *latest* IN delivery is reopened, so partial-return histories may need manual cleanup.

#### Kit "Not Taken" at pickup (customer declines an accessory)

A customer may not take a particular kit/accessory bundled with a unit. Pickup lets the operator mark that kit **"Tidak diambil"** (Not Taken), and it then auto-disappears from the Return checklist — there's nothing to check back in.

- **Storage** — boolean column `delivery_items.not_taken` (migration `2026_06_18_000000…`, on `DeliveryItem` `$fillable`+cast). It is **per-delivery-row**, set **only on the OUT delivery's kit rows**. The flag is deliberately NOT on `UnitKit` (catalog-level, reused across every future rental) nor `RentalItemKit` — it's a property of *this pickup occurrence*. **Kit rows only** (`rental_item_kit_id` not null); the parent unit row can never be "not taken".
- **Pickup UI** ([`PickupOperation`](app/Filament/Resources/Rentals/Pages/PickupOperation.php) + [`pickup-operation.blade.php`](resources/views/filament/resources/rentals/pages/pickup-operation.blade.php)): `markNotTaken($id)` sets `not_taken=true, is_checked=true, condition=null` (flipping `is_checked` so it no longer blocks the `allItemsChecked()` / `validatePickup()` gate — treated as "resolved one way"), and logs to `activity_log` via `Rental::logActivity(..., 'general')`. `undoNotTaken($id)` reverses it (only valid pre-validation; once pickup is validated the OUT delivery is locked like everything else). The kit row renders muted/dashed (`.row.not-taken` CSS) with a "Tidak diambil" pill + "Diambil" undo link; the **"Tidak diambil"** button appears in the row actions for any kit that isn't yet flagged.
- **Why it vanishes at Return** — `Rental::createDeliveries()`'s IN branch (`app/Models/Rental.php`) reads the OUT delivery's `not_taken` kit ids and **(a) deletes any already-created IN row for them** (the IN row is normally created at pickup-mount, *before* the flag is set) **and (b) `continue`s past creating new ones**. With no IN `DeliveryItem` row, `ProcessReturn`'s `getDeliveryItems()`, `scannableList()`, `allItemsChecked()`, blade rows, and `validateReturn()` loop all skip the kit naturally — **no Return-side code changes were needed**. This re-applies on every `createDeliveries()` call (so partial-return cycles stay correct).
- **Scanner guards** — `PickupOperation::scannableList()` rejects `not_taken` kits (alongside `auto_scan_with_parent`); `scanByCode()` and `markAllChecked()` also skip them so a stray scan/mark doesn't re-activate a declined kit.
- **No double-booking impact** — availability/conflict checks are **unit-level** (`ProductUnit`), never kit-level, so a not-taken kit has zero effect on scheduling. The whole unit is still picked up and unavailable for the period as normal. `RentalItemKit.is_returned` stays `false` for a not-taken kit (it was never out, so never returned) — this is intentional and read by nothing that gates completion.
- **OUT delivery note PDF** ([`partials/delivery-note-body.blade.php`](resources/views/pdf/partials/delivery-note-body.blade.php)) shows a grey **"Tidak diambil"** badge for these kit rows instead of the misleading green ✓.

### Unit Scanner + QR/Barcode label system (Pickup & Return)

The scan button on both operation pages ("Scan unit to hand over" / "Scan returned unit") opens a **real camera scanner** that decodes a unit/kit's QR or Code128 barcode and checks the matching item — identical effect to the manual Check button. Recreated from a React design handoff as an Alpine component (the design source lives in the Claude design-extract bundle under `scanner/`, not in the repo). Refer to this whole feature as the **"unit scanner"** (or "scanner popup" / "QR-barcode label system").

- **Closed-system code contract** — [`app/Services/UnitCodeService.php`](app/Services/UnitCodeService.php): every label encodes the string `PREFIX:<serial>` where `PREFIX` = first 4 alphanumeric chars of the company name (`Setting::get('site_name')`, uppercased; falls back to `GEAR`). `decode()` strips the prefix and returns the serial, or `null` for foreign/public codes (scanner ignores them). QR **and** barcode encode the *same* string so reprints are byte-identical. No URLs — a phone camera just shows plain text, nothing opens.
- **Label image generation** — [`app/Services/LabelImageService.php`](app/Services/LabelImageService.php): `png(string $serial, 'qr'|'barcode'): string` returns raw PNG bytes with the serial stamped beneath the code. **Pure GD** (QR drawn by hand from BaconQrCode's `Encoder::encode()->getMatrix()`; barcode via `picqer/php-barcode-generator`'s `BarcodeGeneratorPNG`) — deliberately avoids the imagick extension that `simple-qrcode`'s PNG backend would require. Needs `ext-gd` (present on the server, **absent on the local XAMPP CLI** so it can't be exercised locally).
- **Label download** — route `GET /admin/unit-label/{serial}/{type}` (`admin.unit-label`, `auth`-gated, inside the installed-gate `else` branch in `routes/web.php`) streams the PNG as a download. Exposed via a **Label** action on the product-unit table ([`UnitsRelationManager`](app/Filament/Resources/Products/RelationManagers/UnitsRelationManager.php) → `recordActions`) whose modal ([`resources/views/filament/resources/products/unit-label-modal.blade.php`](resources/views/filament/resources/products/unit-label-modal.blade.php)) lists the parent unit + each serial-bearing kit, each with `[QR]` / `[Barcode]` download links.
- **Kit "auto-scan with parent"** — `unit_kits.auto_scan_with_parent` (boolean, migration `2026_06_07_000000…`, on `UnitKit` `$fillable`+cast, toggle in the `kits` Repeater). ### Bluetooth Label Printer (LuckPrinter) + Print Label page

A **"Print Label"** admin page prints labels directly from the browser to **Luck Jingle** Bluetooth label printers (seri L10/L12/L13/L15/C16/MPL11, 96 dots / 12 mm + a few wider models) via **Web Bluetooth** — no Android app, no driver, no print server. This is the on-device counterpart to the server-rendered PNG download in the [unit scanner / label system](#unit-scanner--qrbarcode-label-system-pickup--return); both encode the **same closed-system payload** so printed labels stay scannable.

- **The driver + editor** — `public/vendor/luckprinter/**`: a standalone, **zero-dependency ES-module** bundle (reverse-engineered from the "Luck Jingle" app, GATT service `0xFF00` + ESC/POS `GS v 0` raster with credit flow-control). Public transport API in `/vendor/luckprinter/luckprinter.js` (`connectLuckPrinter`, `printRaster`/`buildRaster`, `LuckPrinter` EventTarget). Canvas composition in `label.js` (`createLabelCanvas`/`drawText`/`drawQR`/`drawBarcode`/`drawImage`); BLE transport in `driver.js`; model registry in `devices.js`; pure-JS 1D barcode (CODE128/EAN-13) in `vendor/barcode.js`. The real UI is **`editor.html` + `editor-app.js`** — a full **WYSIWYG label designer** (free canvas: add/drag/resize/rotate text·QR·barcode·image·line·rect, per-element property panel, **server-saved templates** (see below — the old hardcoded "Teks+QR/Barcode/Harga/Alamat" quick-template buttons were removed; a fresh open with no queue starts with an empty canvas), undo/redo, save/load to localStorage + export/import JSON, paper-size + orientation + zoom, "Pratinjau Cetak" of the rotated head bitmap). All coordinates in **dots** (203 dpi = 8 dots/mm); the design is landscape and rotated 90° at print time so width = printer head. `test.html` is an older single-label reference UI. **Everything is loaded as raw `<script type="module">`, NOT via Vite** — deliberately zero-build (do not move into `resources/js/`). Folder was relocated `public/luckprinter/` → `public/vendor/luckprinter/`; keep all import paths `/vendor/luckprinter/...`.
- **The page is a dedicated full-screen route, NOT a Filament page / iframe** — the editor ships its own global CSS (`body`/`header` selectors) and a vanilla-JS app bound to fixed DOM ids + `DOMContentLoaded`, so it can't live inside the Filament panel chrome (an earlier `<iframe>` embed inside a Filament page rendered but its controls were dead). Instead, route **`admin.print-label`** (`GET /admin/print-label`, a closure in `routes/web.php` inside the installed-gate `else` branch, `auth`-gated) **serves `public/vendor/luckprinter/editor.html` as its own HTML document**, str-replacing `</head>` to inject `<script>window.LUCKPRINTER_DATA_URL=…</script>` (the units feed path). `editor.html`'s module `src` is **absolute** (`/vendor/luckprinter/editor-app.js`) so it loads correctly when served from this `/admin/...` path. The Filament page [`LabelPrinter`](app/Filament/Pages/LabelPrinter.php) (slug `label-printer`, nav **Inventory**) exists **only as the sidebar menu item**: `getNavigationUrl()` returns `route('admin.print-label')` and `mount()` redirects its own URL there. Do **not** try to port `editor-app.js` into Alpine/Filament markup; edit it in place.
- **Prefill + bulk print via server-injected queue** — `admin.print-label` resolves `?unit={id}` / `?units=1,2,3` **server-side** (DB + auth on hand) into an ordered queue `[{serial,name,payload,type}]` (parent unit + each serial-bearing kit, **`payload = UnitCodeService::encode($serial)`**) and injects it as `window.LUCKPRINTER_QUEUE` — **no client fetch**, so the design prefills reliably. On open `editor-app.js` applies the **default server template** if one exists (`window.LUCKPRINTER_DEFAULT_TEMPLATE` → `deserialize()`, restoring its logo + paper size), otherwise falls back to a hardcoded `insertUnitLabel(queue[0])`, and shows an **"Antrian Cetak"** panel. Elements carry a **`bind`** role (`name` / `serial` / `payload`); **"Cetak semua antrian"** (`printQueue`) loops the queue and, per item, re-renders the *same design* with bound elements swapped (`bindElement`) → one design, N labels (static parts like the logo stay, name/serial/QR change per unit). **`ensureQueueBindings()`** is a safety net: when a default template has no `payload`-bound (or name/serial) element, it auto-binds the first QR/barcode → payload and first/second text → name/serial so per-unit labels never come out identical. A **queue preview navigator** (`‹ i/N ›` floating bar below the canvas, `previewQueueItem()`) lets you step through each label; clicking an item in the "Antrian Cetak" panel jumps to it.
- **Import from system (manual search)** — for free designs, route **`admin.label-printer.units`** (sibling closure, `auth`-gated) returns `{data:[{serial,name,payload,type}]}` filtered by `?q=` (search) or `?ids=` (explicit). `editor-app.js` reads its URL from `window.LUCKPRINTER_DATA_URL` (injected) or `?dataUrl=`, then `fetch`es same-origin. A **"Data Sistem"** panel button + an **"Ambil dari sistem"** button inside the QR/Barcode property panels open `openSystemPicker`; picking sets the element's `data` to the payload and marks `bind:'payload'` (`applyUnitToElement`), or inserts a full unit label. **Never put a raw serial in QR/barcode `data` — always the encoded payload** so it reads on the Pickup/Return scanner.
- **Entry points** — (a) the **Label** modal on the product-unit table ([`unit-label-modal.blade.php`](resources/views/filament/resources/products/unit-label-modal.blade.php)) — a **"Print via Bluetooth"** button → `route('admin.print-label', ['unit' => $record->id])` (new tab; the old QR/Barcode PNG download links stay, for paper printers/archive); (b) a **"Print Labels"** bulk action on [`UnitsRelationManager`](app/Filament/Resources/Products/RelationManagers/UnitsRelationManager.php) → `route('admin.print-label', ['units' => $ids])` (all selected units merged into one queue).
- **System logos** — `admin.print-label` also injects `window.LUCKPRINTER_LOGOS` `[{name,url}]` from settings (`site_logo`/`logo`/`doc_logo`/`pwa_admin_icon`, via `Storage::url`) + each `Brand.logo`. A **"Logo"** toolbar button (hidden when the list is empty) drops the chosen logo onto the canvas (picker when >1). Logos are same-origin (`/storage/...`) so the print canvas isn't tainted.
- **Print geometry + calibration** — the design is landscape (X=length, Y=label width); `makePrintCanvas` crops the centre **`printWidthDots`** strip (96 dots = 12 mm for L12; label paper is 14 mm so 1 mm is non-printable each side) and rotates 90° so width = head. The print sequence in `driver.js` (`enable → wakeup → GS v 0 raster → GS FF → ESC J 40 → stop`) is **byte-identical to the official L12 SDK** (`L12.printTagOnce`), and `mmToDots`/`vMargin` are correct for every size — so a consistent physical offset (commonly ~5 mm) is **printer gap-sensor-to-head calibration, not a layout bug**. The editor has a **"Kalibrasi Posisi Cetak"** control (`state.calib` `{x:length, y:width}` mm, persisted in `localStorage` `luckjingle_print_calib`) that nudges content within the print bitmap (`makePrintCanvas` offsets the crop); it's visible in the **Pratinjau Cetak** preview. Leave margins so the nudge doesn't clip.
- **Editor UX notes** — image elements default to **dithering OFF** (Floyd–Steinberg turns logos/line-art into noise at the small dot resolution — "pecah"; toggle it on per-image for photos). Canvas is **pannable** (drag empty area); zoom has a **Fit-to-page** button + auto-fits on load and on label-size change (slider min `0.25`, fractional). Cable-flag labels render **Depan + Belakang contiguous on the left with the wrap deadzone at the far right**.
- **Server-saved templates (custom, shared)** — table `label_templates` + model [`LabelTemplate`](app/Models/LabelTemplate.php) (`name`, `design` JSON = the editor `serialize()` output incl. `label` paper size, `is_default` — only one default at a time via `setAsDefault()` in a transaction). CRUD routes mirror the calib endpoint (auth-gated, CSRF via `window.LUCKPRINTER_CSRF`): `admin.print-label.templates.index` (GET), `.store` (POST create/update), `.destroy` (DELETE). All templates + the default's design are injected as `window.LUCKPRINTER_TEMPLATES` / `LUCKPRINTER_DEFAULT_TEMPLATE` / `LUCKPRINTER_TEMPLATES_URL`. Sidebar section **"Template Tersimpan (Server)"** lists each with its **paper size badge** (so "beda layout = beda template" is visible) and Muat / Jadikan Default / Hapus; **"Simpan sebagai Template…"** prompts for a name. `localStorage` save/load + export/import JSON are kept alongside for flexibility. A **"Data terikat"** `<select>` in the property panel sets an element's `bind` role (text → none/name/serial; QR/barcode → none/payload) so manually-built templates swap content per unit.
- **QR rendering is dynamic, not snapped** — `drawQR` (in `label.js`) renders the QR module-sharp to a 1px-per-module offscreen canvas then scales it **nearest-neighbor** to fill the requested box exactly. Previously the size snapped to integer multiples of the module count, so resizing "jumped"; now it tracks the element box smoothly while staying crisp/scannable on thermal print.
- **Calibration sync** — the editor's dashed **"area aman cetak"** / non-print overlay stays *inside* the label (it represents the printer's print band); the effect of the calibration nudge is shown in **Pratinjau Cetak** (which is centered + scaled to its panel), not in the editor guide, so the box never drifts off-canvas. Calib persists to DB (`admin.print-label.calib` → `luckprinter_calib_x/y`) with `localStorage` fallback.
- **Icons** — editor UI emoji were replaced with inline stroke SVGs via an `ic(name)` helper (`editor-app.js`) + inline `<svg>` in `editor.html` (and a data-URI SVG for the locked-element badge). The **log box** and the **browser-support banner** keep their text emoji because they're written via `textContent` (SVG can't render there).
- **Constraints** — Web Bluetooth needs **Chrome/Edge** on Android/Windows/Mac and **HTTPS or localhost**; it does **not** exist on iOS/Safari (the editor shows a banner and disables Connect, never errors). `URL::forceScheme('https')` is on, but plain-HTTP local XAMPP still can't connect — test over `https://`/`localhost`. Rendering is client-side canvas (unlike server-GD `LabelImageService`), so the designer works on the local CLI-less environment; only the actual BLE print needs hardware.

### Promotions and discounts

A rental can carry **four** independent discount layers, stacked in this order: (1) **customer-category** discount (the user's `CustomerCategory.discount_percentage`); (2) `DailyDiscount` — "rent N days pay M"; (3) `DatePromotion` — special-date percentage on the amount after daily; (4) manual/coupon `Discount`. `PromotionService::calculatePromotions()` computes layers 2–4 on the **post-category base**; the admin UI for creating the promos lives at `app/Filament/Pages/Promotions.php`.

- **Each layer is stored separately on the `Rental` row** so it's independently auditable: `category_discount_amount` + `category_name` (migration `2026_06_16_000000…`), `daily_discount_id`/`daily_discount_amount`, `date_promotion_id`/`date_promotion_amount`, and `discount`/`discount_type`/`discount_id`/`discount_code` (manual **or** coupon — they share the `discount` column and are mutually exclusive). `Rental.subtotal` is the **gross/real price**; every discount is a line *below* it.
- **The category discount is NOT baked into prices.** Historically `CartController::add()` reduced each item's `daily_rate` by the category %, so a 100% category customer saw every product at Rp 0 with no audit trail. Now the cart stores the **gross** rate and the category discount is applied as an explicit layer at checkout/edit time. `CheckoutController` (`index`/`process`/`validateDiscount`) computes `categoryDiscountAmount = gross × pct`, persists it, and writes `RentalItem.daily_rate`/`subtotal` gross. **When touching cart/checkout pricing, never re-bake the category discount into the rate.**
- **`Rental::discountBreakdown(): array`** ([app/Models/Rental.php](app/Models/Rental.php)) is the single source of truth for rendering — returns ordered `['key','label','amount']` per non-zero layer, with promo names read live via the `dailyDiscount`/`datePromotion`/`discountRelation` relations (generic fallback labels if a promo row was deleted) and the snapshotted `category_name`. Reused by the admin **View Rental** page, the **rental editor** ringkasan, the storefront **checkout/cart/rental-detail/success** views, and the **quotation/invoice PDFs**. Don't re-derive discount lines inline — call this helper.
- **`RentalObserver::recalculateTotals()` folds `category_discount_amount` into the discount total** alongside manual+daily+date; `taxableAmount = subtotal − (all four)`. The total math lives there — don't recompute totals manually after touching discount fields (see the `$touches` note above).
- **Admin can pick promos in the editor ringkasan.** The custom Livewire `RentalEditor` (NOT a Filament form) has a **"Diskon dari Promosi"** picker (desktop + mobile blade) with three `wire:model.live` selects (Daily / Date / Coupon, populated by the `dailyDiscountOptions`/`datePromotionOptions`/`couponOptions` computeds — active promos, listed regardless of date/usage eligibility). Picked amounts are **live-computed** by the `appliedDiscounts` computed from each promo's own rule against the current subtotal/days (category-aware base; daily rate is post-category so re-saving a storefront rental doesn't drift), shown via `promoBreakdown`, and persisted by the shared `discountFields()` helper used by both `save()` and `persistInline()`. Selecting a **coupon overrides the free manual discount** (both occupy `discount`); the `updatedDiscountId` hook syncs `discount_code`. Admin-applied coupons do **not** increment `Discount.usage_count` (only storefront checkout does). Legacy rentals created before the un-baking keep their old net item rates and have null category fields → they simply show no category line (no backfill; original gross is unrecoverable).

### Responsive admin navigation (iPad orientation)

The admin panel switches between a **desktop sidebar** layout and a **mobile-style compact** layout **based on viewport orientation/width, client-side, with no page reload** — so rotating an iPad flips the layout instantly (and in-progress forms like the RentalEditor are preserved). Portrait = compact/mobile; landscape (≥1024px) = sidebar. Driven by **one body class `gr-compact`**, NOT by re-rendering — Filament renders sidebar-vs-top-nav as different server-side DOM, so we commit to one DOM (sidebar) and restyle it.

- **Why not User-Agent**: iPad detection by UA is unreliable — since iPadOS 13 Safari defaults to a *desktop* (macOS) UA, so `/iPad/i` often never matches. All switching is therefore viewport-driven.
- **Server side** — [`AdminPanelProvider`](app/Providers/Filament/AdminPanelProvider.php): `isPhone()` (renamed from `isMobileDevice()`) matches **phones only** and early-returns `false` for `iPad`/tablet UAs. `$useTopNav = ($navigation_layout === 'top' || $isPhone)`. So phones keep forced top-nav (unchanged); tablets + desktop render the sidebar DOM. The QR-scanner render-hook ternary still resolves to `panels::sidebar.footer` for sidebar DOM (the QR float lives inside the sidebar).
- **The class toggle** — [`resources/views/filament/hooks/responsive-navigation.blade.php`](resources/views/filament/hooks/responsive-navigation.blade.php) `<script>`: `isCompact = matchMedia('(orientation: portrait)').matches || innerWidth < 1024` → toggles `body.gr-compact`. Listens to `resize`, `orientationchange`, `matchMedia` `change`, and **`livewire:navigated`** (Filament SPA nav can morph `<body>` and drop the class). **Orientation must beat width**: the iPad Pro 12.9"/13" in portrait is exactly 1024px, which a width-only rule would mis-classify as desktop. (`is-mobile-view` is still toggled at `<768` for back-compat but nothing reads it.)
- **Compact-mode CSS** — `body.gr-compact` rules in [`resources/css/filament/admin/theme.css`](resources/css/filament/admin/theme.css) (appended section): collapse the sidebar out of flow (`position:fixed; width:0`) so content goes full-width, hide `.fi-sidebar-nav`/`.fi-sidebar-header`, hide the global QR float and breadcrumbs. theme.css still **re-anchors the "dynamic island" footer to a fixed top-right account pill**, but the nav hook now **overrides it to `display:none` in compact** — on portrait tablets the pill rendered as a squished capsule stuck at the old (width:0) sidebar edge, so it was dropped. To replace it, the `.gr-bottombar` **"More" menu has an account + Logout block at its foot** (user name/email + a POST form to `route('filament.admin.auth.logout')`, styled via `.gr-bb-more-user`/`.gr-bb-more-logout`). Full Filament **notifications + profile** are still only reachable by rotating to landscape (the sidebar returns). These selectors are no-ops on the phone top-nav DOM (no `.fi-sidebar`), so phones are unaffected. The bottom tab bar (`.gr-bottombar`, same blade) is shown via `body.gr-compact` (was a `@media max-width:767px`) and is the **only** portrait navigation — its "More" menu was expanded to cover all destinations (Finance, Computers, Promotions, Maintenance, Settings, …) since there is no off-canvas drawer.
  - **Why the override lives in the nav hook, not theme.css**: theme.css is **Vite-compiled** (the repo commits built assets + manifest) so editing it needs a rebuild, which can't run in the CLI-less env. The nav-hook blade is an **inline `<style>` at `panels::body.end`** — it renders after the head stylesheet, so an equal-specificity `!important` rule there beats theme.css with **no build**. Prefer it for compact-chrome CSS tweaks. The footer-hide selectors mirror theme.css's exactly (incl. the `:not(.fi-body-has-topbar)` one) so specificity matches and source order decides.
- **Gotchas**: (a) Full-width reclaim depends on Filament 4's flex layout; a defensive `body.gr-compact .fi-main-ctn { width:100%; padding/margin-inline-start:0 }` neutralises any sidebar-width offset — verify in-browser if a left gap appears. (b) The mobile **calendar** `@media (max-width:767px)` block was deliberately left as-is (NOT generalized to `gr-compact`) so portrait tablets keep the native FullCalendar toolbar. (c) Desktop browsers narrowed below 1024px also go compact — expected responsive behaviour. (d) `navigation_layout='top'` users keep top-nav DOM with no orientation behaviour (desktops don't rotate).

#### Responsive breakpoints — the two numbers that matter

There are exactly **two** breakpoints in the admin responsive system, and they mean different things. Don't conflate them, and prefer the `gr-compact` class over re-deriving width in new code:

- **1024px (+ orientation) → `gr-compact` on/off.** The master switch (JS, [responsive-navigation.blade.php](resources/views/filament/hooks/responsive-navigation.blade.php)): `isCompact = matchMedia('(orientation: portrait)').matches || innerWidth < 1024`. So **compact** = portrait (any width) **OR** width < 1024px; **expanded sidebar** = landscape **AND** width ≥ 1024px. This is the canonical "is the chrome in mobile mode" signal — the collapsed sidebar, the `.gr-bottombar`, and the (now hidden) account pill are all gated on `body.gr-compact`. **New responsive code should key off `body.gr-compact`, not a raw `@media` width**, so it tracks the actual chrome state (this is why the rental view/editor + pickup/return pages now hide the bar via `body.gr-compact`, not `@media (max-width:767px)` — the old media query never matched portrait iPad, which is 768–1024px, so the bar leaked through).
- **The page mobile LAYOUT mirrors that condition as a media query.** The custom pages with a distinct phone layout — `view-rental`, `RentalEditor` (its `.desktop-view`/`.mobile-view` toggle + mobile-subhead + modal→bottom-sheet rules), and the `pickup`/`return` operation blades (the comprehensive ≈680px phone reflow incl. `.mobile-actionbar`) — gate that layout with **`@media (max-width: 1023px), (orientation: portrait)`**, which is the literal CSS twin of the `gr-compact` JS test. So portrait iPad now gets the *full* phone layout, not just a hidden bar. These were previously `@media (max-width: 767px/768px/680px)` and never fired on portrait tablet. **Small-phone fine-tuning breakpoints (380/460/520/560/720/760/860px) were left as-is** — they refine within their own narrow range (or are overridden by the structural block) and don't need to fire on tablet. A landscape desktop window < 1024px also gets the phone layout now (consistent with the compact chrome it already had).
- **768px → phone vs. tablet *within* compact.** A second, CSS-only breakpoint distinguishing the two devices that both get `gr-compact`: **phones < 768px**, **portrait tablets 768–1024px**. Used to give iPad a slightly larger bottom bar than phones via `@media (min-width: 768px) { body.gr-compact .gr-bottombar { … } }` (height 76px vs 64px, bigger icons/labels/FAB, content padding 84px vs 72px). Caveat: a large **phone in landscape** (≥768px wide, still compact since <1024px) also matches this and gets the tablet sizing — acceptable edge case; tighten to `(min-width:768px) and (orientation:portrait)` if it ever matters.
- **Per-page bottom-bar suppression** — `view-rental`, `RentalEditor` (create/edit), and the `pickup`/`return` operation blades hide `.gr-bottombar` while open (their own sticky action bar replaces it — "Opsi B"). Each does `body.gr-compact .gr-bottombar { display:none !important; }` plus reclaims the nav hook's reserved bottom padding with **doubled-class** selectors (`body.gr-compact .fi-page.fi-page { padding-bottom:0 !important }`) — the doubling bumps specificity above the nav hook's `body.gr-compact .fi-page` rule, which would otherwise win because the hook renders later at `panels::body.end`.

## Things that regularly surprise

- **`composer update` / `composer install` triggers `filament:upgrade`** (post-autoload-dump). If Filament ships a breaking change this will run during any dep install.
- **Observers registered in `AppServiceProvider`** — easy to miss when tracing rental/finance side-effects.
- **Rental editor is NOT a Filament form** — Create/Edit rental render Livewire `App\Livewire\Admin\RentalEditor` via custom Blade view. `ItemsRelationManager` exists but is unrendered. New rental UI features go in the Livewire component + its Blade, not in Filament forms/relation managers.
- **Don't manually recalc rental totals after touching `RentalItem`** — `RentalItem::$touches = ['rental']` already triggers `RentalObserver::updated` which does full subtotal/discount/tax/total recalc (with `updateQuietly`). Manual `$rental->save()` after item changes will double-process and may overwrite the correct numbers. For the source rental in a cross-rental MOVE (where the item is no longer touching it), use `$rental->touch()` explicitly.
- **The `$isInstalled` gate wraps the entire routes file** in one big `if/else` — adding routes means putting them inside the `else` branch.
- **Duplicate stubs for debugging** live in the repo root (`check_classes_v2.php`, `check_finance_data.php`, `check_schema.php`, `debug_serial.php`, `test_decode.php`). These are throwaway scripts, not part of the app — don't import from them.
- **`layouts/guest.blade.php` is dual-mode**: it renders both `{{ $slot ?? '' }}` (component-style, e.g. `<x-guest-layout>` from `auth/*.blade.php`) AND `@yield('content')` (extends-style, used by `frontend/computers/mobile-kiosk-register*.blade.php`). Don't "clean it up" to one or the other without checking both consumer styles.
- **Carbon 3 `diffInSeconds` returns signed**: `$now->diffInSeconds($earlierDate)` returns NEGATIVE because the parameter is "from" not "to". Several places in the codebase historically used this incorrectly with unsigned-int columns (e.g. `actual_duration_seconds`). Fix is `max(0, (int) abs(...))`. Watch for this when reading old code.
- **`ProductUnitObserver` syncs kit serials**: when a `ProductUnit.serial_number` changes, `ProductUnitObserver::updated` silently (`updateQuietly`) updates all `UnitKit` rows that have `linked_unit_id = this unit` to carry the new serial. If you rename a serial via raw SQL and skip the observer, `KitUnitLinker` will fail to resolve that kit slot on the next save and may spawn a duplicate ghost unit.
- **`UnitKitObserver` guards self-reference**: `UnitKitObserver::saving` throws `DomainException` if a kit slot's serial matches its own parent unit's serial, or belongs to another unit of the same product. This guard fires on all save paths, including console/seeders.
- **Only completed/cancelled rentals are locked from editing**: `Rental::canBeEdited()` returns **true** for `QUOTATION` / `CONFIRMED` / `LATE_PICKUP` / `ACTIVE` / `LATE_RETURN` / `PARTIAL_RETURN` / `EXPIRED`, and **false** only for `COMPLETED` / `CANCELLED`. `EXPIRED` is editable on purpose (so a timed-out quote can be revived), even though it's terminal for availability/checklist. `EditRental::mount()` redirects away when false. `Rental::reopenFromCompleted()` flips `COMPLETED → ACTIVE` precisely to re-enable editing + a redo of the return.
- **`quotation` does NOT auto-become `late_pickup`**: only a `confirmed` rental that overruns its `start_date` is late; an un-confirmed quotation that overruns its `start_date` goes to `expired` (dead-end). The trigger lives in three synced places — see the auto late/expiry transitions note in *The Rental domain* section. Don't re-introduce a `quotation → late_pickup` shortcut.
- **Invoice `late_fee` / `total` are derived, not authoritative**: `Invoice::recalculate()` overwrites them from its rentals (`late_fee = Σ rentals.late_fee`, `total = Σ rentals.total`). Writing a fee onto an `Invoice` row and then calling `recalculate()` silently discards it — always set `rental.late_fee` (+ `recalculateTotal()`) first, then `recalculate()` the invoice. Use `Rental::syncOutstandingInvoice()` instead of hand-rolling this. A late fee never auto-marks anything paid; it only raises the balance to collect via Accounts Receivable.
- **Kiosk register page polling** (`checkin-register.blade.php`): polls `/kiosk/checkin/{slug}/status` (JSON endpoint returning `{has_active_booking, booking_id}`) every 5s. Only redirects when an active checked-in booking exists — not on every poll. The earlier version unconditionally redirected on each tick which caused the kiosk to bounce back to home every 8s. `ComputerCheckinController::show()` also auto-redirects to `kiosk.timer` when an `ACTIVE` + `checked_in_at` booking exists, so the post-mobile-register transition is seamless.
